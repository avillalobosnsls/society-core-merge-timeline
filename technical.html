Technical timeline — the plumbing

The capabilities that have to be true under the hood for the merge to work: identity, lifecycle, visibility, data, and admin control — each with where it stands, when it's needed, and what's still to define.

Overview Phases Technical
How to read this page: ✅ = built and live · ⏳ = in flight with an owner · ❓ = an open question that needs a decision and a design before its "needed by" release. The ❓ items are the ones to pressure-test with Dev.
✅ Live

One login / identity

Needed by: v1.1 · in place since June

Why it matters: every doorway in the merge assumes one account — a member crossing from Core to Society (or back) must stay signed in, on every surface.

Where it stands

  • Centralized auth live across Core, Society, and the app since June
  • Session continuity works app ↔ web — validated before v1 shipped

Watch items

  • Auth context through edge flows (e.g., password reset mid-journey) — fixes shipped with the July auth release
⏳ In flight

Account lifecycle

Needed by: v1.1

Why it matters: when a member deletes or disables their account, every connected system — Core, Society, downstream data — has to hear about it and act.

Where it stands

  • In-app account deletion released, with a 30-day change-of-mind window
  • Deleting the account revokes access to all connected apps
  • Lifecycle webhook (disable/delete events) is live

Still to do

  • Downstream systems register to receive lifecycle events and mark records accordingly (Dev)
❓ To define

Login & access visibility

Needed by: v1.1 launch

Why it matters: the moment the Society door opens we'll want to know it's working — who logged in, who tapped Explore Society, from which surface, and whether they came back.

Open questions

  • Which events do we capture — logins, entry-point taps, Society sessions, return visits?
  • Where does the event data land, and who can query it?
  • Do we need this instrumented before the v1.1 release, or can it follow fast?

Owner

  • Dev + Product — define the event set; Product confirms what questions it must answer
❓ To define

Masquerade / impersonation

Needed by: v1.5

Why it matters: support and admins troubleshoot by seeing what the member sees. Once Core and Society share one account, masquerade either works across both — or support goes blind on half the experience.

Open questions

  • Will masquerade carry across Core and Society under one login?
  • Who is allowed to masquerade, and is every use logged/audited?
  • What does the member-facing record show after a support session?

Owner

  • Dev + Product — needs a decision and a design before v1.5 support load arrives
⏳ In flight

Data mapping

Needed by: v1.5

Why it matters: one member, one record — Core member data, Society profiles and track progress, and the System of Record have to agree on who's who, or metrics and reporting can't be trusted.

Where it stands

  • Member data migration into the System of Record is underway

Still to define

  • How Society track/profile data maps back to the member record
  • Which system is the source of truth for each field once the merge is live

Owner

  • Dev
❓ To define

Metrics & reports

Needed by: v1.5 launch

Why it matters: the launch campaign needs to be measurable from day one — and leadership will ask how the merge is doing in week one, not month three.

Open questions

  • The metric set — entry-point taps, track starts, track completions, return rate?
  • Where reports live and who owns the dashboard
  • Campaign attribution — can Marketing see which content drove entries?

Owner

  • Product + Marketing — define the questions; Dev wires the data
❓ To define

Admin access

Needed by: v1.5

Why it matters: once one account spans both systems, "who can see and manage what" has to be answered once, coherently — not per system.

Open questions

  • Which roles exist across Core + Society, and what can each see or change?
  • How do chapter-level roles (advisors, leaders) map onto Society content?
  • Is admin activity across member data audited in one place?

Owner

  • Dev + Product
